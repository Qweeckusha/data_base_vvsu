## Тема 7. Распределённые базы данных и системы клиент-сервер

### 🔹 Архитектура «клиент–сервер»

Это **модель взаимодействия**, в которой:
- **Клиент** — приложение, которое запрашивает данные (например, веб-сайт, мобильное приложение)
- **Сервер** — СУБД, которая хранит данные и обрабатывает запросы

> 💡 Пример:  
> Ты заходишь в Telegram → клиент отправляет запрос → сервер БД возвращает твои сообщения.

**Минимально необходимые компоненты** (см. вопрос №15 в тесте):  
✅ **Сервер СУБД + клиентское приложение**  
❌ Файловый сервер **не нужен** — это архитектура устаревших файл-серверных систем!

---

### 🔸 Основные принципы клиент-серверных систем

1. **Разделение ролей**: клиент = интерфейс, сервер = логика + данные  
2. **Масштабируемость**: можно добавлять клиентов без изменения сервера  
3. **Централизованное управление данными**: все данные в одном месте → проще обеспечивать целостность и безопасность  
4. **Сетевая передача**: клиент и сервер могут быть на разных машинах

**Критерии оценки** таких систем:
- Производительность
- Надёжность
- Безопасность
- Поддержка параллелизма
- Удобство администрирования

---

### 🔹 Стандарты архитектуры клиент-сервер

- **ODBC (Open Database Connectivity)** — стандартный API для доступа к СУБД (независимо от производителя)
- **JDBC** — Java-аналог ODBC
- **ADO.NET** — для .NET-приложений

> 💡 Эти стандарты позволяют писать приложения, которые работают с **любой СУБД** (MySQL, PostgreSQL, Oracle) без переписывания кода.

---

### 🔹 Промежуточный слой (Middleware)

Иногда между клиентом и СУБД ставят **дополнительный слой**:
- **Сервер приложений** (Application Server) — обрабатывает бизнес-логику
- **TP-мониторы** (Transaction Processing Monitors) — координируют распределённые транзакции (см. Тему 5)

> 📌 Пример:  
> Клиент → Django/FastAPI (сервер приложений) → MySQL (СУБД)  
> Это **трёхзвенная архитектура**: клиент – промежуточный слой – БД

---

### 🔹 Распределённые базы данных (РБД)

**Определение**: БД, данные которой **физически размещены на нескольких узлах** (серверах), но логически воспринимаются как единое целое.

> 💡 Пример:  
> Данные пользователей из Европы хранятся в Германии, из Азии — во Владивостоке, но приложение работает с ними как с одной БД.

---

### 🔸 Основные принципы распределённой обработки

1. **Прозрачность расположения**  
   → Приложение не знает, где физически хранятся данные (вопрос №16 в тесте)

2. **Прозрачность фрагментации**  
   → Данные могут быть разбиты на части (например, по регионам), но запрос `SELECT *` возвращает всё как обычно

3. **Прозрачность тиражирования (репликации)**  
   → Копии данных могут храниться на нескольких узлах для отказоустойчивости

4. **Локальная автономия**  
   → Каждый узел может работать независимо при отсутствии связи

---

### 🔹 Модели распределённых БД

| Тип | Описание |
|-----|--------|
| **Гомогенная** | Все узлы используют **одну и ту же СУБД** (например, только MySQL) |
| **Гетерогенная** | Узлы используют **разные СУБД** (MySQL + Oracle + PostgreSQL) — сложнее интегрировать |

---

### 🔹 Методы построения РБД

1. **Фрагментация (разбиение)**  
   - **Горизонтальная**: строки распределяются по узлам (например, пользователи 1–1000 → сервер A, 1001–2000 → сервер B)  
   - **Вертикальная**: столбцы распределяются (например, `id, имя` → сервер A, `email, телефон` → сервер B)

2. **Репликация (тиражирование)**  
   - Полная: все узлы хранят копию всей БД  
   - Частичная: только часть таблиц реплицируется

---

### 🔹 Проблемы распределённых систем (подтема для СРС!)

1. **Сетевые задержки и отказы**  
   → Запрос может зависнуть или потеряться

2. **Несогласованность данных**  
   → Из-за асинхронной репликации один клиент видит старые данные, другой — новые

3. **Сложность транзакций**  
   → Нужен **двухфазный коммит (2PC)**, который медленный и уязвим к сбоям

4. **Сложность оптимизации запросов**  
   → Планировщик должен решать, **где** выполнять JOIN, чтобы минимизировать передачу данных

5. **Безопасность**  
   → Данные передаются по сети → нужны шифрование и аутентификация на каждом узле

---

### 🔹 Зачем это важно тебе как программисту?

- Современные приложения почти всегда **распределённые** (микросервисы, облака)
- Ты будешь проектировать системы, где БД — не на localhost, а в облаке (AWS RDS, Yandex Managed PostgreSQL)
- Знание проблем РБД помогает выбирать правильную архитектуру: когда использовать репликацию, когда — шардирование
