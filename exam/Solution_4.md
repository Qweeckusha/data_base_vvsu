## Тема 4. Даталогическое проектирование

### 🔹 Что это вообще такое?
**Даталогическое проектирование** — это этап, на котором **концептуальную модель (ER-диаграмму)** превращают в **логическую структуру**, подходящую для конкретной модели данных — чаще всего **реляционной** (таблицы, ключи, связи).

> 💡 Просто:  
> - Концептуальная модель = «что есть в системе» (сущности, связи)  
> - Даталогическая модель = «как это будет выглядеть в виде таблиц»

---

### 🔸 Основные шаги даталогического проектирования

1. **Определение состава БД**  
   — Какие сущности и связи из ER-модели станут таблицами?  
   — Какие атрибуты пойдут в какие таблицы?

2. **Выбор модели данных**  
   В курсе почти всегда — **реляционная модель** (MySQL, PostgreSQL и т.д.).

3. **Применение правил отображения** (см. ниже)

4. **Описание схемы на SQL**  
   — `CREATE TABLE ...`  
   — `PRIMARY KEY`, `FOREIGN KEY`, `CHECK`, домены и т.п.

---

### 🔹 Правила отображения ER-модели в реляционную

| Элемент ER-модели | → | В реляционной модели |
|-------------------|---|------------------------|
| **Сущность** | → | **Таблица** |
| **Атрибут сущности** | → | **Столбец таблицы** |
| **Первичный ключ сущности** | → | `PRIMARY KEY` |
| **Связь "один-ко-многим" (1:N)** | → | В дочерней таблице добавляется **внешний ключ** на родительскую |
| **Связь "многие-ко-многим" (M:N)** | → | Создаётся **отдельная связующая таблица** с двумя FK |
| **Слабая сущность** | → | Таблица с **составным PK**, включающим FK на владельца |

> ✅ Пример M:N:  
> Сущности `Студент` и `Курс` связаны «многие-ко-многим» → создаём таблицу `Зачисление(Студент_ID, Курс_ID)`.

---

### 🔹 Домены
**Домен** — это **тип данных + ограничения** на значения.  
В SQL это реализуется через:
- `INT`, `VARCHAR(50)`, `DATE` — базовые типы
- `CHECK (возраст >= 16)` — дополнительные правила
- Иногда через `ENUM` или пользовательские типы (в PostgreSQL)

> 💡 Важно: домен ≠ просто тип. Это **смысловое ограничение**: например, «курс — целое число от 1 до 6».

---

### 🔹 Критерии оценки базы данных
По РПД, при даталогическом проектировании оценивают:
1. **Нормализованность** — нет ли избыточности (см. Тему 3)
2. **Целостность** — правильно ли заданы PK/FK, `NOT NULL`, `UNIQUE`
3. **Производительность** — учтены ли будущие запросы (например, нужны ли индексы?)
4. **Гибкость** — можно ли расширять модель без переделки
5. **Соответствие требованиям** — все ли данные из предметной области учтены

> 📌 На экзамене могут спросить: *«Какие критерии используются при оценке качества логической модели?»*

---

### 🔹 Пример: от ER к SQL

**ER-модель**:  
- Сущность `Автор(ID, Имя)`  
- Сущность `Книга(ISBN, Название)`  
- Связь M:N: «Автор пишет Книгу»

**Логическая модель**:
```sql
CREATE TABLE Автор (
    ID INT PRIMARY KEY,
    Имя VARCHAR(100) NOT NULL
);

CREATE TABLE Книга (
    ISBN CHAR(17) PRIMARY KEY,
    Название VARCHAR(255) NOT NULL
);

CREATE TABLE Автор_Книга (
    Автор_ID INT,
    Книга_ISBN CHAR(17),
    PRIMARY KEY (Автор_ID, Книга_ISBN),
    FOREIGN KEY (Автор_ID) REFERENCES Автор(ID),
    FOREIGN KEY (Книга_ISBN) REFERENCES Книга(ISBN)
);
```

> 💡 Обрати внимание:  
> - Связь M:N → отдельная таблица  
> - Составной PK в связующей таблице  
> - FK ссылаются на PK родительских таблиц

---

### 🔹 Зачем это важно тебе как программисту?
- Ты будешь **проектировать схемы БД** для своих проектов (Telegram-боты, веб-приложения)
- ORM (Django, SQLAlchemy) всё равно генерируют SQL — лучше понимать, **что происходит под капотом**
- При оптимизации запросов знание логической структуры критично
- На собеседованиях часто дают задачу: «спроектируйте БД для X» — это и есть даталогическое проектирование
