## Тема 6. Целостность и безопасность данных

### 🔹 Что такое целостность?
**Целостность** — это свойство БД, при котором **все данные соответствуют заданным правилам и ограничениям**.  
Если целостность нарушена — данные становятся **некорректными или противоречивыми**.

> 💡 Пример: если в таблице `Заказы` есть запись с `Клиент_ID = 999`, а в таблице `Клиенты` такого ID нет — нарушена **ссылочная целостность**.

---

### 🔸 Виды ограничений целостности

| Тип | Описание | Пример в SQL |
|-----|--------|--------------|
| **Сущностная** | Первичный ключ не может быть NULL и должен быть уникальным | `PRIMARY KEY`, `UNIQUE NOT NULL` |
| **Доменная** | Значение должно соответствовать типу и диапазону | `CHECK (возраст >= 18)`, `VARCHAR(50)` |
| **Ссылочная** | Внешний ключ должен ссылаться на существующий PK | `FOREIGN KEY (Группа_ID) REFERENCES Группы(ID)` |
| **Пользовательская** | Бизнес-правила, не покрываемые стандартными средствами | «Студент не может быть старше 40 лет» → `CHECK (возраст <= 40)` |

---

### 🔹 Декларативная vs процедурная целостность

| Подход | Как реализуется | Плюсы / Минусы |
|--------|------------------|----------------|
| **Декларативная** | Через `CONSTRAINT`: `PRIMARY KEY`, `FOREIGN KEY`, `CHECK`, `UNIQUE` | ✅ Просто, быстро, выполняется на уровне СУБД<br>❌ Ограниченные возможности |
| **Процедурная** | Через триггеры, хранимые процедуры | ✅ Гибкость (можно писать сложную логику)<br>❌ Сложнее отлаживать, медленнее |

> 💡 Пример:  
> - Декларативно: `CHECK (баланс >= 0)`  
> - Процедурно: триггер, который при попытке перевода денег проверяет, не превысит ли сумма лимит за день

---

### 🔹 Безопасность базы данных

**Цель безопасности** — обеспечить:
- **Конфиденциальность**: только авторизованные пользователи видят данные
- **Целостность**: данные не искажаются несанкционированно
- **Доступность**: данные доступны авторизованным пользователям, когда нужно

---

### 🔸 Модели безопасности

1. **Простейшая модель**  
   - Есть пользователи и роли  
   - Каждому назначаются **привилегии** (`SELECT`, `INSERT`, `UPDATE`, `DELETE`)  
   - Пример: студент может читать таблицу `Расписание`, но не редактировать

2. **Многоуровневая модель** (например, Bell–LaPadula)  
   - Данные и пользователи имеют **уровни секретности** («секретно», «совершенно секретно» и т.д.)  
   - Пользователь может читать данные **не выше своего уровня**, писать — **не ниже**  
   - Используется в военных и госсистемах

> 📌 В твоём курсе акцент на **простейшую модель** — она реализуется через SQL.

---

### 🔹 Управление доступом в SQL

```sql
-- Создание пользователя
CREATE USER 'студент'@'localhost' IDENTIFIED BY 'пароль';

-- Назначение прав
GRANT SELECT ON университет.* TO 'студент'@'localhost';

-- Отзыв прав
REVOKE INSERT, UPDATE ON университет.оценки FROM 'студент'@'localhost';

-- Создание представления для ограничения данных
CREATE VIEW оценки_студента AS
SELECT * FROM оценки WHERE студент_id = USER_ID();

GRANT SELECT ON оценки_студента TO 'студент'@'localhost';
```

> 💡 Представления (`VIEW`) — мощный инструмент: можно показывать **только нужные строки/столбцы**, скрыв остальное.

---

### 🔹 Контроль параллельной обработки
Это часть **целостности при одновременной работе**:
- Используются **блокировки** и **уровни изоляции транзакций** (см. Тему 5)
- Цель — предотвратить **грязное чтение**, **фантомы** и другие аномалии

### 🔹 Обслуживание и восстановление БД

##### Источники сбоев:
- Аппаратные (падение диска, отключение питания)
- Программные (ошибки в коде, баги СУБД)
- Человеческий фактор (случайное `DROP TABLE`)

#### Резервное копирование:
| Тип | Описание |
|------|--------|
| **Полное** | Копия всей БД |
| **Инкрементальное** | Только изменения с момента последнего бэкапа |
| **Логическое** | SQL-дамп (`mysqldump`) |
| **Физическое** | Копия файлов данных |

#### Процедуры восстановления:
1. **ROLLBACK** — откат незавершённых транзакций (через журнал транзакций)
2. **Восстановление из бэкапа** + **накат логов** (до точки сбоя)

> 💡 MySQL использует **журнал двоичных логов (binlog)** для восстановления после сбоя.

### 🔹 Подтема для самостоятельного изучения (из РПД):
> **Сравнение логической и физической модели баз данных**

| Аспект | Логическая модель | Физическая модель |
|--------|------------------|-------------------|
| **Что описывает** | Таблицы, столбцы, связи, ключи | Файлы, индексы, партиционирование, кэши |
| **Зависит от СУБД?** | Нет (абстрактная) | Да (конкретная реализация) |
| **Кто работает** | Проектировщик БД | Администратор БД (DBA) |
| **Пример** | `Студент(ID, Имя, Группа_ID)` | InnoDB-таблица в файле `student.ibd`, B+ дерево индекса по `ID` |

> ✅ На экзамене могут спросить: *«Чем отличается логическая модель от физической?»*
